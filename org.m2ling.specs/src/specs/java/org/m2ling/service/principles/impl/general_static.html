m2ling /  m2principles

Notes

  • Views may or not have a project-grain. They can represent only a project sub-system or on the contrary shows a set of projects.
  • A concept item namespace is equal to the viewpoint name.
  • A viewpoint can content component groups from others VP or the same VP (read only). The bound type is given by the "boundType" reference. If a component is used in another VP, it can't be removed. If enumeration maps a component-type group, values are those of its immediate sub-component. If a bound type is given but none enumeration, the possible values are all component of given bound type in every view of the bound VP. If a component is binded and has a void name, it get the bound component name.
  • Bound type : none, component type without components as enum, component type with component group(s) as enum.
  • A component engine type can be from another VP
  • If an enum is given but no bound type, same bound type than own component type is assumed
  • If a component type is bound to another type, each component nodes of one of its component must be bound to one of the bound type nodes (if any) associated with the bound component.
  • The abstraction level is relative and is selected by the end-user according to its needs in its viewpoint. For instance, it is possible to declare :
    • Component type (M2) : JEE application Server
    • Component (M1) : JBoss
    • Component node (M0) : a running JBoss 5 instance
    • Component type (M2) : Application Server
    • Component (M1) : JEE AS
    • Component node (M0) : a running JEE AS
    • Component type (M2) : JBoss 5 application Server
    • Component (M1) : Any JBoss 5.0.3 instance
    • Component node (M0) : a running JBoss 5.0.3 instance
    • Component type (M2) : JBoss 5 application Server
    • Component (M1) : Any JBoss 5.0.3 instance
  • Instantiation factor enforces the number of possibles component nodes for a component of a given type.
