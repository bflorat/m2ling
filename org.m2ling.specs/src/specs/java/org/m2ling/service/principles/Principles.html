    • Notes

    • Views may or not have a project-grain. They can represent only a project sub-system or on the contrary shows a set of projects.
    • The abstraction level is relative and is selected by the end-user according to its needs in its viewpoint. For instance, it is possible to declare :
      • Component type (M2) : JEE application Server
      • Component (M1) : JBoss
      • Component instance (M0) : a running JBoss 5 instance
      • Component type (M2) : Application Server
      • Component (M1) : JEE AS
      • Component instance (M0) : a running JEE AS
      • Component type (M2) : JBoss 5 application Server
      • Component (M1) : Any JBoss 5.0.3 instance
      • Component instance (M0) : a running JBoss 5.0.3 instance
      • Component type (M2) : JBoss 5 application Server
      • Component (M1) : Any JBoss 5.0.3 instance
    • A concept item namespace is equal to the viewpoint ID.
    • A viewpoint can contain component groups from others VP or the same VP (read only in both cases). The bound type is given by the "boundType" reference.
    • It is not possible to drop a component, component instance or associated component type while bound from another VP.
    • If a component is binded and has a void name, it get the bound component name.
    • If a component type is bound to another type, each component instances of one of its component must be bound to one of the bound type instances (if any) associated with the bound component.
      Decisions
    • Component type's enumeration maps ArchitectureItem, not directly ComponentTYpe to allow setting both ComponentType and ComponentTypeGroup items. Others types should be rejected.
