Component creation

Administrators or actors with adequate authorizations can create a new component

Nominal rules

See common service requirements

A new component created with Roundtrip
  • View id=id_view_vp_logical_Logical_BikesOnline
  • ctID=id_ct_logical_servicecontainer
  • id=a_new_id
  • name=Weblogic
  • description=Another service container
  • comment=Runs some cool services
  • tags=logical,soa
  • bound id=id_comp_tech_weblogic
  • references ids=RUNS:id_comp_view_Logical_BikesOnline_AdminServices;DEPENDS_ON:id_comp_view_Logical_BikesOnline_PostgreSQL,id_comp_view_Logical_BikesOnline_AuthenticationLDAP
  • status=APPLICABLE
should have following characteristics (full round trip from presentation to presentation) : boundComponent=id=id_comp_tech_weblogic name=Weblogic comment=Runs some cool services description=Another service container drop= id=a_new_id name=Weblogic references=[targets=[id=id_comp_view_Logical_BikesOnline_AdminServices name=AdminServices ] type=RUNS , targets=[id=id_comp_view_Logical_BikesOnline_PostgreSQL name=PostgreSQL , id=id_comp_view_Logical_BikesOnline_AuthenticationLDAP name=AuthenticationLDAP ] type=DEPENDS_ON ] status=APPLICABLE tags=logical,soa type=id=id_ct_logical_servicecontainer name=ServicesContainer

References rules

#COMP1 : All references defined in CT has not to exist at the component level but those who exist must follow types defined in their associated CT.

In this case, the CT creation should : FAIL with code NONE_TARGET

#COMP3 : A reference target can't be null.

In this case, the CT creation should : FAIL with code TARGET_NOT_FOUND

Binding rules

#COMP2 : If a CT is bound, all components of its type should be binded to a component of the bound CT.

If reference is null, the component creation should : FAIL with code NULL_ARGUMENT
